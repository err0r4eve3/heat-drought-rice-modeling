# 项目 Codex 指令

## 基本约束

- 用户回复使用简体中文，代码、命令、路径和包名保持英文。
- 当前项目只支持 Windows 环境，优先使用 Python。
- 不做 Web 应用或花哨前端。
- 修改代码后必须运行基础测试，默认使用 `uv run python -m pytest`。
- 不要依赖全局 `py` 环境；项目测试和 pipeline 应通过仓库内 `.venv` 执行。
- 不要删除已有数据。
- 不要修改 `data/raw/` 中的原始数据；原始数据只读。
- 所有处理结果写入 `data/interim/`、`data/processed/` 或 `data/outputs/`。
- 不要提交大文件、原始数据、虚拟环境、缓存或生成的大型结果。

## 脚本规则

- 每个 `scripts/` 脚本必须支持命令行运行。
- 每个脚本必须有日志，优先使用 `loguru`。
- 每一步失败时应记录 warning 或 error；数据缺失不能中断整个项目，除非该步骤的核心输入完全不可用。
- 所有输出文件写入前必须创建父目录。

## 空间与大数据处理

- 任何空间数据处理都必须显式记录 CRS。
- 默认 CRS 为 `EPSG:4326`，面积统计使用 `EPSG:6933`。
- 大数据操作优先使用 chunk、`dask`、`xarray`、`zarr`、`parquet` 或流式读取。
- 不要把大栅格或大表强行展开成巨大 CSV。

## 验证

- 真实数据不存在时，测试应使用临时模拟数据。
- 修改 `src/` 或 `scripts/` 后，优先运行相关测试，再运行受影响脚本。
- 如果验证无法运行，最终回复必须说明原因和未验证风险。

## NotebookLM 引用流程

- NotebookLM 用于项目事实、外部数据源、文献引用和阶段性总结；当前同步源清单见 `docs/notebooklm-workflow.md`。
- 后续需要引用外部数据源或生成研究摘要时，优先查询项目 NotebookLM 笔记本，再回到原始 URL 或本地报告核验关键事实。
- NotebookLM 输出不能替代本地代码、测试和生成结果；实现行为以仓库文件为准。
- 不要向 NotebookLM 上传密钥、账号、cookies、私有原始数据或敏感信息。
